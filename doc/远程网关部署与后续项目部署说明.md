## 0. 当前状态更新（2026-04-17）

本说明最初只覆盖“远端网关 + 本地业务服务联调”。

当前已经新增一套独立的远端中间件栈，包含：

- MySQL
- PostgreSQL
- Redis
- RabbitMQ
- Nacos

该中间件栈与 `remote-gateway` 分目录、分 Compose 项目、分网络管理，不和现有远端网关或服务器上其他服务混在一起。

中间件的当前部署与后续接入方式，统一以 `doc/远程中间件部署说明.md` 为准。

## 1. 当前部署目标

当前方案不是把整套后端都部署到服务器，而是：

- 服务器只部署 `gateway-service`
- 本地继续运行业务服务：
  - `auth-service`
  - `master-service`
  - `academic-service`
  - `workflow-service`
  - `billing-service`
  - `notify-service`
- 前端本地开发环境直接请求远端网关
- 远端网关通过 SSH 反向隧道回调本地业务服务

这个方案适合：

- 只想把“中间层/网关”先放到服务器
- 本地继续调试各业务服务
- 不希望把数据库、Redis、MQ、Nacos 一起搬到服务器

## 2. 当前服务器情况

### 2.1 跳板机

- 作用：作为 SSH 跳板机，连接目标服务器
- 当前项目已经验证可通过跳板机访问目标机
- 凭据和密钥不要写入仓库文档，后续继续沿用现有终端工具或本机私密配置

### 2.2 目标服务器

- 系统：Ubuntu 24.04.2 LTS
- 已安装：
  - Docker 29.3.0
  - Docker Compose v5.1.0

### 2.3 当前远端部署目录

- 目录：`/opt/school-erp/stacks/remote-gateway`

该目录下当前只放四类文件：

- `app.jar`
- `Dockerfile`
- `docker-compose.remote.yml`
- `.env`

### 2.4 当前远端 Compose 项目

- Compose Project Name：`school-erp-remote-gateway`
- 网络：独立 bridge 网络
- 容器：只包含 `gateway-service`

这意味着：

- 不会和服务器里其他 Docker 服务混在同一个 Compose 项目里
- 不会把你的其他本地业务服务也部署到服务器
- 后续要继续部署别的项目，可以再建新的独立目录和独立 Compose 项目

## 3. 当前端口规划

### 3.1 对外服务端口

- `18080`：远端 `gateway-service` 对外访问端口

这是目前真正暴露给前端或调用方使用的远端端口。

### 3.2 SSH 反向隧道端口

目标服务器上还会监听以下端口：

- `18081` -> 本地 `8081` (`auth-service`)
- `18082` -> 本地 `8082` (`master-service`)
- `18083` -> 本地 `8083` (`academic-service`)
- `18084` -> 本地 `8084` (`workflow-service`)
- `18085` -> 本地 `8085` (`billing-service`)
- `18086` -> 本地 `8086` (`notify-service`)

注意：

- 这些不是部署在目标机上的业务服务
- 它们只是 SSH 反向隧道入口
- 请求最终仍然回到你本机的服务进程

## 4. 当前联调拓扑

```text
浏览器
  -> 本地前端 http://localhost:5173
  -> 远端网关 http://36.137.84.162:18080
  -> 目标机上的隧道端口 18081~18086
  -> 回到本机 8081~8086
  -> 本地各业务服务响应
```

## 5. 当前仓库内相关文件

### 5.1 网关配置

- `backend/gateway-service/src/main/resources/application.yml`

用途：

- 网关路由改成环境变量驱动
- 不再写死下游服务地址
- 可通过 `.env` 指向本地、远端、隧道地址

### 5.2 远端容器编排

- `backend/gateway-service/docker-compose.remote.yml`

用途：

- 定义远端 `gateway-service` 容器
- 绑定 `18080:8080`
- 通过 `host.docker.internal` 回调目标机宿主上的反向隧道端口

### 5.3 远端镜像构建

- `backend/gateway-service/Dockerfile`

用途：

- 只接收 `app.jar`
- 不上传整个 backend 源码到服务器
- 部署边界清晰，避免误传其他服务

### 5.4 部署脚本

- `scripts/deploy-remote-gateway-legacy.ps1`

用途：

- 本地只打 `gateway-service` 的 jar
- 上传 `app.jar`、`Dockerfile`、`docker-compose.remote.yml`、`.env`
- 在目标机上构建镜像并启动容器
- 支持跳板机参数

### 5.5 SSH 隧道脚本

- `scripts/start-remote-gateway-tunnels.ps1`
- `scripts/paramiko-reverse-tunnel.py`

用途：

- 建立目标机 `18081~18086` 到本地 `8081~8086` 的反向映射
- `paramiko-reverse-tunnel.py` 是当前已验证可工作的稳定实现

### 5.6 前端联调配置

- `frontend/.env.local`
- `frontend/.env.remote-gateway.example`

当前方式：

- 前端直接请求远端网关
- 不依赖本地 Vite 代理来转发 API

原因：

- 之前本地 Vite 代理在处理远端网关响应时出现过 HTTP parse error
- 直接请求远端网关更稳定

## 6. 当前推荐使用方式

### 6.1 本地启动

本地启动以下服务：

- `auth-service`
- `master-service`
- `academic-service`
- `workflow-service`
- `billing-service`
- `notify-service`
- `frontend`

不要启动本地 `gateway-service`，避免和远端网关混用。

### 6.2 启动远端网关

如果远端网关镜像或配置有变更，执行：

```powershell
.\scripts\deploy-remote-gateway-legacy.ps1
```

如果需要显式走跳板机：

```powershell
.\scripts\deploy-remote-gateway-legacy.ps1 -JumpHost <bastion-ip> -JumpUser <bastion-user> -JumpPort 22
```

### 6.3 启动反向隧道

优先使用当前已经验证过的 Python 方案，或者沿用现有后台隧道进程。

如果后续要重新做成正式命令，可继续封装 `scripts/paramiko-reverse-tunnel.py`。

### 6.4 前端访问

前端本地地址：

```text
http://localhost:5173
```

前端 API 当前指向：

```text
http://36.137.84.162:18080
```

## 7. 服务器上已经做过的配置变更

当前为了让容器可以访问 SSH 反向隧道端口，目标服务器的 SSH 配置已经允许：

- `AllowTcpForwarding yes`
- `GatewayPorts clientspecified`

这一步是必须的，否则：

- 反向隧道只能绑定回环地址
- 容器就无法通过宿主机端口访问本地服务

## 8. 后续继续部署项目时的建议

### 8.1 不要把所有项目堆到同一个 Compose 里

建议按项目拆目录，例如：

```text
/opt/school-erp/stacks/remote-gateway
/opt/school-erp/stacks/erp-backend-full
/opt/school-erp/stacks/erp-frontend
/opt/school-erp/stacks/another-project
```

每个目录：

- 一个独立 `docker-compose.yml`
- 一个独立 `.env`
- 一个独立 project name

这样做的好处：

- 容器命名清晰
- 网络边界清晰
- 回滚方便
- 不会误重启其他项目

### 8.2 统一规划端口

建议把服务器端口按项目分段：

- `18080~18089`：当前远程联调网关/辅助端口
- `18100~18199`：后续完整 ERP 后端
- `18200~18299`：后续前端/静态资源/Nginx

不要直接抢占：

- `80`
- `443`
- `8080`

除非后续明确要做统一入口层。

### 8.3 生产化时再补基础设施

当前这套联调方案没有真正把以下基础设施部署到服务器：

- PostgreSQL
- Redis
- RabbitMQ
- Nacos

原因是当前代码本身还处于 demo/联调阶段，业务服务没有完整接入这些中间件。

后续如果要把整套 ERP 真正部署上服务器，建议顺序是：

1. 先确定数据库配置
2. 再补 Redis
3. 再补 MQ
4. 最后决定是否需要 Nacos 做注册/配置中心

不要在当前联调阶段一次性把这些都上齐，否则复杂度会明显上升。

### 8.4 不要把凭据写入仓库

仓库文档里不要记录：

- 跳板机密码
- 目标机私钥
- 服务器 root 密码
- `.env` 中的敏感值

建议：

- 密钥只放本机私密目录
- 生产环境变量只放服务器本地 `.env`
- 仓库只保留 `.example` 文件

## 9. 当前方案的优点与限制

### 9.1 优点

- 只部署中间层，变更成本低
- 本地调试体验保留
- 不污染目标服务器现有服务
- 后续容易逐步扩展为更多远端项目

### 9.2 限制

- 本地电脑必须在线，反向隧道必须存活
- 本地服务挂掉，远端网关也会联调失败
- 这不是完整生产部署方案，只是远程联调方案

## 10. 后续建议

如果后面准备把项目进一步上服务器，建议按下面顺序推进：

1. 先把当前远端网关联调方案稳定下来
2. 再决定是否把前端静态资源部署到服务器
3. 再决定是否把 `auth-service` 和 `master-service` 先搬到服务器
4. 等数据库与缓存方案定下来后，再做整套后端远端化
