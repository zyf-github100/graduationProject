# 完整后端微服务部署说明

## 1. 目标

这份说明对应的是“真正以微服务方式运行后端”的部署路径，不再使用当前远端网关回调本地服务的 SSH 反向隧道方案。

部署后，运行形态变为：

- `gateway-service` 作为统一入口
- `auth-service`
- `master-service`
- `academic-service`
- `workflow-service`
- `billing-service`
- `notify-service`

这些服务统一接入：

- Nacos：服务注册发现
- PostgreSQL：各服务独立持久化入口
- Redis：缓存/会话预留
- RabbitMQ：异步事件总线

## 2. 当前仓库已经补齐的能力

当前代码已经新增以下微服务基础能力：

1. 所有服务支持 `microservice` 运行模式。
2. 所有服务可通过 Nacos 注册发现。
3. 网关默认使用 `lb://service-name` 路由，而不是写死 `localhost`。
4. `academic-service` 已增加对 `master-service` 的 Feign 调用示例。
5. `workflow-service` 已增加审批状态事件发布。
6. `notify-service` 已增加 RabbitMQ 监听和事件收件箱接口。
7. 仓库已新增 `backend/docker-compose.microservice.yml`，用于整套后端接入现有 `school-erp-middleware-net`。

## 3. 依赖前提

先确保远端中间件栈已经存在，并保持和文档一致：

- Docker 网络：`school-erp-middleware-net`
- PostgreSQL：`postgres:5432`
- Redis：`redis:6379`
- RabbitMQ：`rabbitmq:5672`
- Nacos：`nacos:8848`

如果这些中间件已经按 [远程中间件部署说明](/D:/graduationProject/doc/远程中间件部署说明.md) 部署完成，就可以直接接入，不需要再单独给后端重新起一套中间件。

## 4. 构建方式

在仓库根目录执行：

```powershell
cd backend
mvn package -DskipTests "-Dapp.finalName=app"
```

这一步会生成各服务的 jar，`docker-compose.microservice.yml` 会直接把这些 jar 打进镜像。

## 5. 环境变量

以 `backend/.env.microservice.example` 为模板，准备一份实际环境文件，例如：

```text
backend/.env.microservice
```

至少需要改这些值：

- `POSTGRES_PASSWORD`
- `REDIS_PASSWORD`
- `RABBITMQ_PASSWORD`
- `NACOS_PASSWORD`

如果 Nacos 还没有推送配置文件，建议先保持：

- `NACOS_CONFIG_ENABLED=false`

先使用仓库内 `application.yml` 作为服务本地配置，等后续把配置迁移到 Nacos 再打开。

## 6. 启动方式

默认推荐直接使用仓库脚本：

```powershell
.\scripts\deploy-remote-backend-full.ps1 -StopLegacyGateway
```

如果跳板机登录改成了私钥认证，并且不是走默认本机 SSH Key，可显式指定：

```powershell
.\scripts\deploy-remote-backend-full.ps1 -JumpPrivateKeyFile <local-key-path> -StopLegacyGateway
```

该脚本会做三件事：

1. 用 `mvn package -DskipTests "-Dapp.finalName=app"` 生成各服务部署 jar。
2. 上传整套后端的 Compose、Dockerfile 和 jar 到远端目录。
3. 默认停止旧的 `school-erp-remote-gateway` legacy 容器，再启动新的完整微服务栈。

当前已落地的远端栈信息如下：

- Compose 项目名：`school-erp-backend-full`
- 远端目录：`/opt/school-erp/stacks/erp-backend-full`
- 对外端口：`18080 -> gateway-service:8080`
- 中间件网络：`school-erp-middleware-net`

如果后续要切到 GitHub Actions 按服务自动发布，参考 [GitHub Actions微服务自动部署说明](/D:/graduationProject/doc/GitHub%20Actions微服务自动部署说明.md)。

### 6.1 本机 Docker

```powershell
cd backend
docker compose --env-file .env.microservice -f docker-compose.microservice.yml up -d --build
```

### 6.2 远端服务器 Docker

推荐目录：

```text
/opt/school-erp/stacks/erp-backend-full
```

建议上传以下文件：

- `backend/docker-compose.microservice.yml`
- `backend/Dockerfile.service`
- 各模块 `target/*.jar`
- 实际 `.env.microservice`

然后在目标机执行：

```bash
docker compose --env-file .env.microservice -f docker-compose.microservice.yml up -d --build
```

## 7. 端口与网络边界

推荐只对外暴露网关：

- `18080 -> gateway-service:8080`

其余服务只加入内部网络：

- `backend-core`
- `school-erp-middleware-net`

也就是说：

1. 浏览器或前端只访问网关。
2. 网关通过服务发现转发给业务服务。
3. 业务服务通过 Docker 网络访问 PostgreSQL / Redis / RabbitMQ / Nacos。
4. 不再需要当前那套 `18081~18086` 的 SSH 反向隧道联调方式。

## 8. 验证项

部署完成后，至少验证这些点：

1. Nacos 控制台能看到 7 个服务实例。
2. 访问网关 `/api/v1/academic/student/home` 时，返回体包含 `profileSnapshot`，说明 `academic-service -> master-service` 调用生效。
3. 调用 `workflow-service` 的审批接口后，`notify-service` 的 `/api/v1/notify/tasks/inbox` 能看到新事件，说明 MQ 链路生效。
4. 前端所有 API 只指向网关，不直接访问业务服务。

## 9. 现阶段仍然保留的缺口

这次升级解决的是“运行形态不是微服务”的问题，但还有两类工作没有一次性做完：

1. 业务数据仍以演示内存数据为主，没有把各服务真正持久化到自己的 PostgreSQL schema。
2. Nacos 配置中心虽然已经预留接入点，但还没有把配置正式迁移到 Nacos Data ID。

也就是说，当前仓库已经从“多模块本地演示”升级为“可按微服务方式运行的后端基础版”，下一步最合适的工作是：

1. 给各服务补独立 schema 与 MyBatis/Mapper 落库。
2. 把公共配置和服务配置逐步迁移到 Nacos。
3. 再补统一监控、日志聚合和自动部署脚本。
