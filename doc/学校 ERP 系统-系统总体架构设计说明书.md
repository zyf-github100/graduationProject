# 1 引言

## 1.1 编写目的

本文档用于说明学校 ERP 系统一期项目的总体架构设计方案，明确系统的建设目标、架构分层、服务拆分、数据组织方式、接口交互机制、部署结构、安全控制、运维治理方案以及后续扩展边界，为项目的开发、测试、实施、上线和运维提供统一依据。

本文档的主要目的包括：

1. 明确系统的整体技术方案和设计边界；
2. 统一项目组对系统架构的理解；
3. 为详细设计、接口设计、数据库设计提供上位依据；
4. 为项目上线、运维和后续扩展提供标准参考。

## 1.2 适用范围

本文档适用于学校 ERP 系统一期建设范围，覆盖以下核心业务域：

1. 统一认证与权限管理；

4. 审批流程管理；
5. 收费管理；

本文档不覆盖具体页面交互细节、不展开到字段级数据库设计、不替代接口明细文档和详细设计文档。

## 1.4 术语说明

### 1.4.1 ERP

企业资源计划系统。在本项目中，指学校核心资源与业务流程的一体化管理平台。

### 1.4.2 主数据

在多个业务模块中被重复引用、需要统一维护的数据，如学校、校区、班级、课程、教师、学生等。

### 1.4.3 微服务

将系统按业务领域拆分为多个边界清晰、可独立部署、可独立扩展的服务单元。

### 1.4.4 数据权限

控制用户在拥有功能权限的前提下，能够查看或操作哪些数据范围。

### 1.4.5 读库

面向查询和统计分析的数据库，一般用于报表、汇总、跨域分析，不直接承载核心交易写入。

# 2 建设目标

## 2.1 业务目标

本系统建设目标是面向学校日常管理业务，建设一套可上线、可运行、可扩展的一期 ERP 平台，实现以下核心业务闭环：

1. 学校组织、学期、班级、课程、教师、学生、家长等基础档案的统一管理；
2. 教务核心业务在线化，包括排课、课表、考勤、成绩；
3. 审批流程在线化，包括请假、调课、报修等；
4. 收费管理在线化，包括费用项目、账单、收款、退费、对账；
5. 消息通知能力统一，包括站内信和第三方消息通道；
6. 关键操作留痕，支持审计、报表、导入导出。

## 2.2 技术目标

系统在技术上应达到以下目标：

1. 支持业务服务独立开发与独立部署；
2. 支持按业务域逐步扩展二期模块；
3. 支持统一认证、统一授权、统一审计；
4. 支持与外部系统对接；
5. 支持日志、监控、告警、链路追踪；
6. 支持灰度发布、版本回滚和备份恢复；
7. 支持生产环境稳定运行。

## 2.3 架构目标

本项目架构设计强调以下原则：

1. 以业务边界为核心，而不是为拆分而拆分；
2. 以半年可上线为第一约束，避免过度设计；
3. 以稳定性、可维护性、可扩展性为主要目标；
4. 以权限、数据、流程、运维四个方面为设计重点。

## 2.4 非目标范围

为保证一期项目按期上线，以下内容不纳入本次架构重点建设范围：

1. 通用低代码流程引擎；
2. 原生移动 App；
3. 自定义 BI 平台；
4. 复杂多租户 SaaS 收费体系；
5. 服务网格等高复杂度基础设施；
6. 宿舍、图书、资产、人事、食堂等二期扩展模块。

# 3 总体设计原则

## 3.1 业务边界清晰原则

系统按核心业务领域划分服务边界，不允许多个服务交叉维护同一类核心业务数据。
 主数据由基础数据服务统一维护，教务、收费、审批分别维护各自交易数据。

## 3.2 简化优先原则

一期建设强调“中颗粒度微服务”，不追求服务数量多，而追求服务边界稳、交付节奏稳、联调复杂度可控。

## 3.3 最终一致性原则

单服务内部保证本地事务一致性；跨服务交互采用“同步接口 + 异步事件”的组合模式，通过消息驱动和补偿机制实现最终一致。

## 3.4 安全优先原则

系统涉及学生、家长、教师等敏感信息，必须从设计阶段纳入认证、授权、审计、脱敏、加密、导出控制、备份恢复等安全能力。

## 3.5 可观测性原则

系统必须具备日志、监控、告警和链路追踪能力，保证问题可发现、可定位、可恢复。

## 3.6 可扩展性原则

一期完成后，应支持平滑扩展二期模块，例如宿舍、图书、资产、人事、选课等，不推翻现有架构。

------

# 4 系统总体架构设计

## 4.1 总体架构概述

本系统采用“分层架构 + 微服务架构 + 事件驱动补充”的总体设计模式。
 整体分为以下六层：

1. 终端访问层；
2. 接入与安全层；
3. 业务服务层；
4. 平台能力层；
5. 数据存储层；
6. 运维治理层。

## 4.3 各层功能说明

### 4.3.1 终端访问层

用于承载用户访问入口，一期以管理后台 Web 和教师端 H5/响应式页面为主。

### 4.3.2 接入与安全层

用于统一处理请求接入、认证校验、路由分发、限流、灰度、统一异常等。

### 4.3.3 业务服务层

按业务域提供核心能力，是系统的主干层。

### 4.3.4 平台能力层

为业务服务提供公共支撑，如缓存、消息、文件、配置、任务调度等。

### 4.3.5 数据存储层

负责主数据、交易数据、缓存数据、文件数据、报表数据和日志数据的持久化存储。

### 4.3.6 运维治理层

负责代码发布、环境管理、监控告警、日志管理、链路追踪、备份和恢复。

# 5 系统分层架构说明

## 5.1 终端访问层设计

### 5.1.1 管理后台

服务对象包括学校管理员、教务人员、财务人员、班主任、信息中心管理员等。
 主要功能特点如下：

1. 配置项多；
2. 表单与列表操作多；
3. 审批与查询场景多；
4. 适合采用 PC Web 形式承载。

### 5.1.2 教师端

服务对象包括任课教师和审批节点处理人。
 主要功能包括：

1. 课表查询；
2. 考勤登记；
3. 待办审批；
4. 消息提醒；
5. 成绩录入。

### 5.1.3 学生/家长端

一期不作为重点建设对象，仅保留未来扩展接口和认证体系兼容能力。

## 5.2 接入与安全层设计

接入与安全层由统一入口组件构成，其职责如下：

1. 统一接收前端请求；
2. 统一做身份认证；
3. 统一做接口鉴权；
4. 统一做路由分发；
5. 统一做访问控制和限流；
6. 统一做请求日志和链路标识注入；
7. 统一输出标准异常码。

该层建议由以下能力组成：

1. Nginx/Ingress；
2. API Gateway；
3. 统一认证中心；
4. 安全拦截机制；
5. 访问日志采集机制。

## 5.3 业务服务层设计

业务服务层按领域进行服务拆分，一期建议控制在 6 个核心服务内，以降低团队协作复杂度和联调成本。

## 5.4 平台能力层设计

平台能力层不承载独立业务流程，但为业务系统提供统一支撑，避免公共逻辑散落到各业务服务中。

## 5.5 数据存储层设计

数据存储层按“主数据、交易数据、缓存数据、文件数据、报表数据、日志数据”分类组织。

## 5.6 运维治理层设计

运维治理层用于保障系统可持续发布、可持续运行和可持续定位问题。

# 6 微服务划分与职责说明

## 6.1 服务划分原则

本项目采用“按领域划分、按边界建模、按职责归属”的服务划分方法。
 服务拆分时遵循以下原则：

1. 一个服务只负责一个核心领域；
2. 一个服务拥有自己的业务数据；
3. 禁止跨服务直接更新数据库；
4. 高频调用的边界要清晰、稳定；
5. 一期不拆分过细，不做微碎服务。

## 6.2 服务清单

一期建议建设以下核心服务：

1. API 网关服务；
2. 认证权限服务；
3. 基础数据服务；
4. 教务服务；
5. 审批服务；
6. 收费服务；
7. 消息通知服务。

其中，API 网关服务属于统一接入层，其余为核心业务服务。

------

## 6.3 API 网关服务

### 6.3.1 服务定位

API 网关作为所有请求的统一入口，负责转发和控制，不承载具体业务规则。

### 6.3.2 核心职责

1. 路由转发；
2. 认证拦截；
3. 权限透传；
4. 请求限流；
5. 灰度控制；
6. 统一异常返回；
7. 请求链路标识透传；
8. 统一日志记录。

### 6.3.3 设计要求

1. 所有外部请求必须经过网关；
2. 网关不直接存储业务数据；
3. 网关需支持高并发和多实例部署；
4. 网关需要支持后续新服务接入。

------

## 6.4 认证权限服务

### 6.4.1 服务定位

认证权限服务负责“谁可以访问系统、谁可以操作什么、谁可以看见哪些数据”。

### 6.4.2 核心职责

1. 用户管理；
2. 角色管理；
3. 菜单权限管理；
4. 按钮权限管理；
5. 数据权限管理；
6. 登录与令牌签发；
7. 登录会话管理；
8. 权限校验支撑；
9. 审计入口对接。

### 6.4.3 核心对象

1. 用户；
2. 角色；
3. 权限；
4. 菜单；
5. 岗位；
6. 数据范围；
7. 登录会话。

### 6.4.4 权限模型

系统权限采用三层模型：

#### 1. 功能权限

控制用户是否有权访问菜单、页面、按钮和接口。

#### 2. 数据权限

控制用户可查看或处理哪些校区、年级、班级、课程、学生等数据。

#### 3. 字段权限

控制某些敏感字段是否展示、是否脱敏、是否允许导出。

### 6.4.5 设计说明

1. 权限控制以后端为准；
2. 前端菜单仅作展示控制，不作为最终安全边界；
3. 敏感操作必须记录审计日志；
4. 导出权限必须单独受控。

------

## 6.5 基础数据服务

### 6.5.1 服务定位

基础数据服务负责维护全系统统一主数据，是各业务服务共同依赖的底层业务支撑服务。

### 6.5.2 核心职责

1. 学校信息维护；
2. 校区维护；
3. 学部/年级/班级维护；
4. 学期维护；
5. 节次与时间段维护；
6. 课程维护；
7. 教室维护；
8. 教师档案维护；
9. 学生档案维护；
10. 家长/监护人档案维护；
11. 字典配置和参数维护；
12. 批量导入导出。

### 6.5.3 设计说明

1. 任何业务服务不得自行维护主档案的主版本；
2. 业务服务只引用主数据标识和必要快照；
3. 主数据变更需要留痕和版本控制；
4. 导入必须支持模板、校验、失败反馈。

### 6.5.4 核心价值

该服务是全系统的“数据地基”。主数据不统一，教务、收费、审批都会失真。

------

## 6.6 教务服务

### 6.6.1 服务定位

教务服务负责学校教学运行中的核心流程，是一期业务价值最高的服务之一。

### 6.6.2 核心职责

1. 教学计划基础维护；
2. 排课规则配置；
3. 课表生成与调整；
4. 班级课表查询；
5. 教师课表查询；
6. 教室课表查询；
7. 学生考勤登记与统计；
8. 成绩录入、审核、发布、统计；
9. 教学日历支撑。

### 6.6.3 内部子模块

1. 排课管理；
2. 课表管理；
3. 考勤管理；
4. 成绩管理。

### 6.6.4 设计说明

1. 教务服务依赖基础数据服务中的学生、教师、课程、教室等主数据；
2. 教务服务接收审批服务发送的“请假通过”“调课通过”等事件；
3. 教务服务可向通知服务发送“成绩发布”“课表变更”等通知请求。

------

## 6.7 审批服务

### 6.7.1 服务定位

审批服务负责学校内部固定业务流程的流转和待办管理。

### 6.7.2 核心职责

1. 请假申请与审批；
2. 调课申请与审批；
3. 报修申请与审批；
4. 待办中心；
5. 已办中心；
6. 抄送中心；
7. 审批意见留痕；
8. 状态流转管理。

### 6.7.3 设计说明

1. 一期采用固定流程模板，不做复杂通用流程引擎；
2. 审批服务负责流程流转，不直接修改教务或收费核心交易表；
3. 审批通过后通过消息事件驱动相关业务服务更新业务状态。

### 6.7.4 状态模型

建议审批流程统一使用以下状态：

1. 草稿；
2. 已提交；
3. 审批中；
4. 已通过；
5. 已驳回；
6. 已撤销；
7. 已关闭。

------

## 6.8 收费服务

### 6.8.1 服务定位

收费服务负责学校收费相关业务的规则管理、账单管理、收款处理和退费处理。

### 6.8.2 核心职责

1. 费用项目配置；
2. 账单生成；
3. 学生应收管理；
4. 收款登记；
5. 欠费跟踪；
6. 退费处理；
7. 对账支撑；
8. 收费导出与留痕。

### 6.8.3 设计说明

1. 费用规则、账单、支付记录、退费记录必须分离；
2. 收费操作必须具备完整审计链；
3. 财务权限必须与普通教学角色严格隔离；
4. 支付结果与账务状态要清晰区分。

### 6.8.4 核心对象

1. 费用项目；
2. 计费规则；
3. 学生账单；
4. 收款记录；
5. 退费记录；
6. 对账记录。

------

## 6.9 消息通知服务

### 6.9.1 服务定位

消息通知服务负责消息统一发送和发送状态管理，不承载具体业务判断。

### 6.9.2 核心职责

1. 站内消息；
2. 模板消息；
3. 审批待办提醒；
4. 账单通知；
5. 成绩发布提醒；
6. 第三方通道发送；
7. 发送日志记录；
8. 重试补偿。

### 6.9.3 设计说明

1. 业务服务只发起通知请求，不直接调用短信或第三方 SDK；
2. 发送过程尽量异步化；
3. 消息发送结果要可查询、可追溯；
4. 失败消息需支持重试和补发。

# 7 平台公共能力设计

## 7.1 缓存能力

缓存用于提升系统查询性能和降低重复计算成本，主要适用场景如下：

1. 登录会话缓存；
2. 用户权限缓存；
3. 主数据热点缓存；
4. 限流计数；
5. 消息幂等键；
6. 短期临时状态。

缓存使用遵循以下原则：

1. 缓存不是数据唯一来源；
2. 核心业务状态不得只存缓存；
3. 缓存必须设置合理过期时间；
4. 缓存刷新策略要清晰。

## 7.2 消息队列能力

消息队列用于服务解耦和异步处理，适用场景包括：

1. 请假审批通过后同步考勤状态；
2. 调课审批通过后同步课表状态；
3. 账单生成后发送提醒；
4. 成绩发布后发送通知；
5. 审计日志异步落库；
6. 导入任务完成后异步通知。

消息事件设计应遵循业务语义清晰原则，例如：

1. LeaveApproved；
2. ScheduleAdjusted；
3. BillGenerated；
4. PaymentCompleted；
5. GradePublished。

## 7.3 文件存储能力

系统中存在大量文件处理场景，包括导入模板、导出报表、附件材料、报修图片等。
 文件能力设计要求如下：

1. 文件元数据和业务记录关联；
2. 文件实体统一进入对象存储；
3. 上传限制文件类型、大小、访问权限；
4. 下载需校验权限；
5. 敏感文件应具备失效控制。

## 7.4 定时任务能力

统一定时任务中心用于处理以下任务：

1. 账单定时生成；
2. 日考勤汇总；
3. 导入结果清理；
4. 报表同步；
5. 消息失败重试；
6. 数据归档。

## 7.5 报表读库能力

为了避免跨服务直接联表影响交易系统性能，系统设置统一报表读库。
 其设计原则如下：

1. 报表读库仅用于查询与统计；
2. 交易数据通过事件或定时同步至读库；
3. 跨域统计统一从读库查询；
4. 报表不直接跨业务库做高频联查。

# 8 数据架构设计

## 8.1 数据分层设计

系统数据分为三层：

### 8.1.1 主数据层

由基础数据服务统一维护，包括学校、班级、课程、教师、学生等基础档案。

### 8.1.2 交易数据层

由各业务服务分别维护本领域的业务过程数据，例如：

1. 教务服务维护课表、考勤、成绩；
2. 审批服务维护申请单、审批记录；
3. 收费服务维护账单、收款、退费。

### 8.1.3 分析数据层

由报表读库负责维护汇总统计数据，用于固定报表和综合分析。

## 8.2 数据隔离原则

系统采用“每服务独立数据边界”的模式。
 执行要求如下：

1. 每个服务拥有自己的数据库或独立 schema；
2. 服务之间不得跨库直接写入数据；
3. 服务间数据交互通过接口或事件完成；
4. 前端不得绕开服务层直接拼接数据库逻辑。

## 8.3 主键设计原则

建议采用技术主键与业务编码分离的设计方式：

1. 技术主键用于系统内部关联；
2. 业务编码用于展示、检索和外部对接。

例如：

- student_id 作为内部关联主键；
- student_no 作为学号；
- teacher_id 作为内部主键；
- teacher_no 作为工号。

## 8.4 一致性策略

### 8.4.1 单服务一致性

单服务内部使用本地事务保证强一致性。

### 8.4.2 跨服务一致性

跨服务不采用重型分布式事务，采用如下模式：

1. 业务发起服务先提交本地事务；
2. 成功后发布业务事件；
3. 下游服务消费事件更新本地状态；
4. 若失败则通过重试、补偿或人工介入修复。

## 8.5 快照设计

为保证历史数据可追溯性，业务表可适当保存主数据快照字段，例如：

1. student_name_snapshot；
2. class_name_snapshot；
3. term_name_snapshot；
4. fee_item_name_snapshot。

快照仅用于历史展示和追溯，不替代主数据主版本。

## 8.6 数据迁移设计

数据迁移采用“批次化、可校验、可追溯”的模式：

1. 导入模板标准化；
2. 上传后生成迁移批次；
3. 执行结构校验；
4. 执行业务校验；
5. 生成错误明细；
6. 业务确认后正式入库；
7. 保留迁移日志和导入结果。

# 9 接口与集成架构设计

## 9.1 内部接口设计

内部服务调用采用如下方式：

1. 同步调用使用 HTTP/REST；
2. 异步通信使用消息队列事件。

## 9.2 接口规范要求

所有接口应统一以下内容：

1. URL 命名规范；
2. 请求方式规范；
3. 认证头规范；
4. 分页规范；
5. 排序规范；
6. 返回体规范；
7. 错误码规范；
8. 幂等控制机制。

## 9.3 幂等要求

以下场景必须保证幂等：

1. 提交审批；
2. 生成账单；
3. 支付回调；
4. 退费处理；
5. 导入任务提交；
6. 消息发送请求。

## 9.4 外部系统集成

一期可能对接的外部系统包括：

1. 短信平台；
2. 企业微信或钉钉；
3. 支付接口或银校接口；
4. 历史数据导入接口；
5. 设备或考勤数据接口。

外部对接采用“集成适配层”封装模式，原则如下：

1. 业务服务不直接耦合第三方 SDK；
2. 第三方差异由适配层屏蔽；
3. 接口异常必须标准化；
4. 对接失败必须有降级方案。

# 10 核心业务数据流设计

## 10.1 登录认证数据流

```
用户登录
→ 网关接收请求
→ 转发至认证权限服务
→ 校验账号、密码、状态
→ 生成令牌与会话
→ 返回前端
→ 前端携带令牌访问业务接口
→ 网关做令牌校验和路由转发
→ 业务服务处理请求
```

### 说明

1. 登录态由认证权限服务统一管理；
2. 网关负责拦截和放行；
3. 用户身份与权限信息可通过令牌和缓存配合传递。

## 10.2 请假审批数据流

```
用户提交请假申请
→ 审批服务保存申请单
→ 生成待办任务
→ 消息服务通知审批人
→ 审批人处理申请
→ 审批服务更新状态为已通过
→ 发布 LeaveApproved 事件
→ 教务服务消费事件并更新考勤状态
→ 消息服务通知申请人
→ 审计服务记录全过程
```

### 说明

1. 审批服务不直接修改教务交易表；
2. 教务服务通过事件接收最终业务结果；
3. 该流程支持补偿和重试。

## 10.3 调课数据流

```
教师或教务发起调课申请
→ 审批服务受理并流转
→ 审批通过
→ 发布 ScheduleAdjusted 事件
→ 教务服务更新课表
→ 消息服务通知相关教师和班级管理员
```

## 10.4 账单生成与收款数据流

```
财务配置费用项目与规则
→ 收费服务生成学生账单
→ 发布 BillGenerated 事件
→ 消息服务发送账单通知
→ 财务登记收款或对接支付回调
→ 收费服务更新支付与账务状态
→ 报表读库同步汇总结果
```

## 10.5 成绩发布数据流

```
教师录入成绩
→ 教务服务保存成绩数据
→ 审核通过
→ 状态变更为已发布
→ 发布 GradePublished 事件
→ 消息服务发送成绩发布通知
→ 报表读库同步成绩统计结果
```

# 12 安全架构设计

## 12.1 身份认证安全

系统应具备以下身份安全控制：

1. 密码加密存储；
2. 首次登录改密；
3. 密码复杂度校验；
4. 登录失败锁定；
5. 可选验证码；
6. 会话失效控制；
7. 强制下线机制。

## 12.2 授权与访问控制

系统应具备以下访问控制能力：

1. 基于角色的访问控制；
2. 基于数据范围的授权；
3. 接口级权限校验；
4. 高风险功能单独授权；
5. 导出权限单独控制；
6. 敏感字段脱敏显示。

## 12.3 数据安全控制

1. 敏感字段脱敏；
2. 关键字段加密存储；
3. 导出记录审计；
4. 数据备份加密；
5. 附件下载权限校验；
6. 日志不可随意篡改。

## 12.4 网络安全控制

1. 通过统一入口访问系统；
2. 网关层配置限流；
3. 后端服务尽量走内网通信；
4. 运维入口采用专用权限控制；
5. 生产环境禁止随意开放数据库访问。

## 12.5 审计要求

以下操作必须纳入审计：

1. 登录与登出；
2. 角色与权限变更；
3. 数据导出；
4. 删除或修改关键数据；
5. 收费与退费操作；
6. 审批通过与驳回动作。

# 14 运维与可观测性设计

## 14.1 日志体系

日志分为以下三类：

### 14.1.1 应用日志

记录程序运行信息、异常信息、调用信息。

### 14.1.2 业务日志

记录关键业务动作，例如审批、收费、导出、成绩发布等。

### 14.1.3 审计日志

记录高风险行为和权限变更行为。

## 14.2 监控体系

系统应监控以下指标：

1. 服务运行状态；
2. CPU、内存、磁盘使用情况；
3. 接口耗时；
4. 错误率；
5. 数据库连接数与慢查询；
6. Redis 命中率；
7. 消息队列堆积情况；
8. 定时任务执行状态。

## 14.3 告警体系

告警按严重程度分级：

1. 一级告警：系统不可用、数据库故障、收费异常；
2. 二级告警：错误率显著升高、消息堆积严重；
3. 三级告警：非关键任务失败、容量预警。

## 14.4 链路追踪

系统需具备全链路追踪能力，至少支持：

1. 请求唯一标识贯穿全链路；
2. 追踪跨服务调用路径；
3. 定位慢调用节点；
4. 将链路标识与日志关联。

## 14.5 发布治理

发布流程至少包括：

1. 构建；
2. 自动化测试；
3. 镜像生成；
4. 测试环境部署；
5. 预发验证；
6. 生产审批；
7. 灰度或全量发布；
8. 冒烟测试；
9. 回滚预案。

# 15 架构边界与约束说明

## 15.1 一期架构边界

一期项目重点解决以下问题：

1. 核心业务主链路可跑通；
2. 权限体系可控；
3. 数据体系统一；
4. 审计和运维可落地；
5. 系统可正式上线。

## 15.2 一期不建议过度建设的内容

1. 通用流程引擎；
2. 自定义 BI 平台；
3. 十多个以上的细粒度微服务；
4. 过重的中间件体系；
5. 多终端全面并行建设；
6. 大规模自定义规则引擎。

## 15.3 架构风险点

### 15.3.1 过度拆分风险

若服务拆分过细，将导致联调成本、接口成本、运维成本大幅上升。

### 15.3.2 主数据不统一风险

若班级、课程、学生等主数据在多个服务中重复维护，将直接导致业务错乱。

### 15.3.3 权限模型失控风险

若未在一期建立清晰的数据权限模型，后续补救成本极高。

### 15.3.4 报表直查交易库风险

若高频报表直接压交易库，会影响线上核心业务。

### 15.3.5 数据迁移后置风险

若数据迁移设计过晚，系统上线前容易集中暴露结构性问题。

------
