# 远程中间件部署说明

## 1. 当前结论

当前已经在目标服务器上新增一套独立的中间件栈，包含：

- MySQL
- PostgreSQL
- Redis
- RabbitMQ
- Nacos

这套栈和远端网关 `school-erp-remote-gateway` 完全隔离：

- 单独目录：`/opt/school-erp/stacks/remote-middleware`
- 单独 Compose Project：`school-erp-remote-middleware`
- 单独 Docker 网络：`school-erp-middleware-net`
- 单独数据卷：`school-erp-middleware-*`

当前没有补 GitHub 推送自动部署。现在采用的是“本机执行部署脚本，经跳板机把文件推到目标机，然后在目标机上用 Docker Compose 启动”的方式。

## 2. 服务器上的实际部署情况

### 2.1 远端目录

目标机目录：

```text
/opt/school-erp/stacks/remote-middleware
```

目录内容包括：

- `docker-compose.yml`
- `.env`
- `mysql/init/*`
- `redis/redis.conf`

### 2.2 Compose 项目

- Project Name：`school-erp-remote-middleware`
- Network Name：`school-erp-middleware-net`

这意味着后续新的后端服务如果也部署到这台服务器，只要加入同一个 Docker 网络，就可以直接通过容器名访问这些中间件。

### 2.3 当前容器

- `school-erp-remote-middleware-mysql-1`
- `school-erp-remote-middleware-postgres-1`
- `school-erp-remote-middleware-redis-1`
- `school-erp-remote-middleware-rabbitmq-1`
- `school-erp-remote-middleware-nacos-1`

## 3. 端口规划

### 3.1 容器内服务地址

如果后续新的 Docker 服务加入 `school-erp-middleware-net`，可以直接使用：

- MySQL：`mysql:3306`
- PostgreSQL：`postgres:5432`
- Redis：`redis:6379`
- RabbitMQ：`rabbitmq:5672`
- Nacos：`nacos:8848`

### 3.2 目标机宿主端口

当前这些中间件的宿主机端口全部只绑定到 `127.0.0.1`，默认不对公网暴露：

- `127.0.0.1:18110` -> MySQL `3306`
- `127.0.0.1:18111` -> PostgreSQL `5432`
- `127.0.0.1:18112` -> Redis `6379`
- `127.0.0.1:18113` -> RabbitMQ `5672`
- `127.0.0.1:18114` -> RabbitMQ 管理台 `15672`
- `127.0.0.1:18115` -> Nacos HTTP `8848`
- `127.0.0.1:18116` -> Nacos gRPC `9848`
- `127.0.0.1:18117` -> Nacos Raft `9849`

这个策略的目的：

- 不把数据库、缓存、注册中心直接暴露到公网
- 让同机部署的应用优先走 Docker 网络访问
- 需要本机调试时，再通过 SSH 隧道或终端工具映射出来

## 4. 仓库内相关文件

### 4.1 编排文件

- `infra/remote-middleware/docker-compose.yml`

用途：

- 定义 5 个中间件服务
- 固定独立网络和命名卷
- 默认只绑定到目标机回环地址

### 4.2 环境变量示例

- `infra/remote-middleware/.env.example`

用途：

- 给出端口和账号字段模板
- 仓库里不保存真实密码

### 4.3 MySQL 初始化脚本

- `infra/remote-middleware/mysql/init/00-create-nacos-db.sh`
- `infra/remote-middleware/mysql/init/20-nacos-schema.sql.txt`
- `infra/remote-middleware/mysql/init/21-import-nacos-schema.sh`
- `infra/remote-middleware/mysql/init/22-seed-nacos-admin.sh`

用途：

- 初始化 Nacos 所需数据库
- 导入 Nacos 官方 MySQL schema
- 创建默认 Nacos 管理员账号

### 4.4 Redis 配置

- `infra/remote-middleware/redis/redis.conf`

### 4.5 部署脚本

- `scripts/deploy-remote-middleware.py`

用途：

- 从本机通过跳板机连接目标机
- 首次部署时自动生成本地临时 `.env`
- 把中间件栈文件上传到目标机
- 在目标机执行 `docker compose up -d`
- 支持 `--reset-volumes` 做重建
- 支持通过 `JUMP_PRIVATE_KEY_FILE` 或 `--jump-private-key-file` 使用跳板机私钥登录

## 5. 凭据与配置文件

### 5.1 真实凭据保存位置

当前真实中间件凭据不写入仓库，只保存在：

- 目标机：`/opt/school-erp/stacks/remote-middleware/.env`
- 本机临时文件：`tmp/remote-middleware.generated.env`

本机 `tmp/` 已加入忽略规则，不应提交。

### 5.2 Nacos 默认控制台账号

当前初始化脚本会创建默认管理员：

- 用户名：`nacos`
- 密码：`nacos`

这个账号只在目标机回环端口上可访问，但后续仍建议改掉。

### 5.3 RabbitMQ / MySQL / PostgreSQL / Redis 密码

这些密码由部署脚本首次生成，并落到本机临时文件和目标机 `.env`。

不要把这些值抄进仓库文档。

## 6. 当前验证结果

已经在目标机完成以下验证：

- MySQL 可启动，并存在 `school_erp` 与 `nacos_config` 数据库
- PostgreSQL 可正常登录 `school_erp`
- Redis 可返回 `PONG`
- RabbitMQ 管理 API 可正常响应
- Nacos 健康检查返回 `OK`
- Nacos 默认管理员 `nacos / nacos` 可成功登录

## 7. 后续业务服务怎么接

### 7.1 如果后续服务也部署到服务器 Docker

推荐做法：

1. 新建独立目录，例如 `/opt/school-erp/stacks/erp-backend-full`
2. 新服务仍使用自己的 `docker-compose.yml`
3. 在新服务的 Compose 里把 `school-erp-middleware-net` 声明为外部网络
4. 直接用容器名连中间件

示例思路：

```yaml
networks:
  middleware-core:
    external: true
    name: school-erp-middleware-net
```

然后应用里直接配：

- MySQL: `mysql:3306`
- PostgreSQL: `postgres:5432`
- Redis: `redis:6379`
- RabbitMQ: `rabbitmq:5672`
- Nacos: `nacos:8848`

### 7.2 如果业务服务暂时还在宿主机或本地开发机

可以走两种方式：

- 目标机本机进程：直接用 `127.0.0.1:18110~18117`
- 本地开发机：通过 SSH 隧道把目标机回环端口映射到本机

## 8. 运维边界

当前这套方案的边界很明确：

- 已经把基础中间件上到服务器
- 还没有把业务服务整体搬上服务器
- 还没有做 GitHub 推送自动部署
- 还没有做统一入口、证书、监控告警、备份策略自动化

也就是说，这是一套“可用的独立中间件底座”，不是完整生产平台。

## 9. 建议的下一步

建议后续按这个顺序推进：

1. 先让业务服务逐步改成真正使用 PostgreSQL / Redis / RabbitMQ / Nacos
2. 再选择 1 到 2 个后端服务先搬到服务器
3. 等依赖关系稳定后，再决定是否补 CI/CD 或 GitHub 自动部署
4. 最后再做统一入口、监控、备份与权限收敛
